主控板：MSPM0G3507 天猛星最小系统板

角色：整个系统的大脑。读传感器 → 算 PID → 控制电机，全在它上面跑。
它是什么？
TI 公司推出的 ARM Cortex-M0+ 芯片，80MHz 主频，带 ADC、定时器（PWM/编码器模式）、I2C、UART 等外设。天猛星是第三方做的最小系统板，把芯片引脚全部引出，方便接线。

为什么选它？ H 题强制要求 TI MSPM0 系列。M0+ 内核够用，DriverLib 库函数写起来比手写寄存器轻松很多。

用它学会什么？ TI MCU 生态（CCS IDE + SysConfig 图形化配置 + DriverLib），以后电赛遇到 TI 芯片的题直接复用。
怎么接线（它跟谁连）
ADC (PA0~PA7) → 8 路灰度传感器
TIMG PWM → 四路编码器电机驱动模块
TIMG 编码器 → 四路编码器电机驱动模块（编码器反馈）
I2C (PB2=SCL, PB3=SDA) → OLED + MPU6050（共用 I2C 总线，地址不同）
UART → JY901（如果需要）/ 串口调试
GPIO → 蜂鸣器 + LED + 启动按键
常见翻车
• 供电不足：天猛星用 USB 供电时 5V 输出电流有限，电机启动瞬间可能拉低电压导致 MCU 复位。解决方案：电机单独用电池供电，MCU 用另一路
• 引脚冲突：SysConfig 配引脚时注意 I2C 和 UART 不要和 ADC/PWM 冲突。先画好引脚分配表再动手
• 下载失败：确认 CCS 里选了正确的调试器（XDS110 / DAP-Link），确认复位线接好了

8 路灰度传感器

角色：车的"眼睛"。唯一能看到黑线的传感器。输出 8 路模拟电压 → ADC 读取 → 算黑线偏移量。
它是什么？
一排 8 个红外对管（红外发射+接收），固定在车前底部。每个对管照地面，黑线吸收红外 → 接收管电流小 → 输出电压低；白布反射红外 → 接收管电流大 → 输出电压高。

为什么 8 路？ 路数越多，黑线位置算得越准。5 路只能区分"偏左/正中/偏右"，8 路能算出精确的偏移量 (-100~+100)，巡线更平滑。

线宽 1.8cm → 黑线同时覆盖 2~3 个传感器，通过加权平均算出黑线中心位置。
代码怎么写
/* mod_grayscale.c — 8 路 ADC → line_pos */ #define GRAY_COUNT 8 int16_t mod_grayscale_read(void) { uint16_t adc[GRAY_COUNT]; // 存 8 路原始 ADC 值 (0~4095) uint8_t bit[GRAY_COUNT]; // 0=白地 1=黑线 int16_t sum = 0; int16_t cnt = 0; for (int i = 0; i < GRAY_COUNT; i++) { adc[i] = bsp_adc_read_ch(i); // 跟校准阈值比：低于阈值 → 黑线 bit[i] = (adc[i] < threshold[i]) ? 1 : 0; } for (int i = 0; i < GRAY_COUNT; i++) { if (bit[i]) { // 加权求和：最左权重 -7，最右权重 +7，中间权重 0 sum += (i * 2 - (GRAY_COUNT - 1)) * 100 / (GRAY_COUNT - 1); cnt++; } } return cnt ? (sum / cnt) : 0; // 8路全白→0；左边黑→负；右边黑→正 } /* 这段代码的效果：黑线在正中间 → line_pos ≈ 0 * 黑线偏右 → line_pos > 0（方向环会让车左转纠正） * 黑线偏左 → line_pos < 0（方向环会让车右转纠正） * 白地（全白）→ line_pos = 0（不纠偏，靠陀螺仪走直线） */
常见翻车
• 阈值设错了：用固定数值在不同场地完全不通用。必须每场现采——车放黑线上读 8 路存 black[]，放白布上读存 white[]，阈值=(black+white)/2
• 传感器高度不对：离地面太远 → 读数模糊（黑白差太小）。推荐 3~5mm，太近了会蹭地面
• 串扰：相邻传感器互相照到。用遮光罩隔开每个传感器
• 环境光：赛场上方的灯管直接照到传感器 → 读数全偏高。加遮光罩 + 动态阈值

四路编码器电机驱动模块

角色：接收 MCU 的 PWM 信号 → 驱动 520 电机转动 + 读取编码器反馈。这个模块替代了 TB6612（电机驱动）+ 独立编码器采集两部分。
它是什么？
一块集成了 H 桥驱动器和编码器接口的板子。MCU 不需要分别接 TB6612 的 IN1/IN2/PWM 和编码器的 A/B 相——这个模块内部处理了。你只需要给两个 PWM 信号，它自己控制电机方向和速度，同时把编码器信号整理好回传给 MCU 的定时器。

四路 = 可以带 4 个电机。我们只用 2 路（左轮+右轮），剩下 2 路备用。

为什么不用 TB6612？ TB6612 只管驱动，编码器还得单独接。四路一体板省了接线 + 编码器信号已经调理好了，MCU 直接读就行。
怎么接线
MCU PWM1 → M1 速度控制 → 520 电机 A（左轮）
MCU PWM2 → M2 速度控制 → 520 电机 B（右轮）
MCU GPIO → M1/M2 方向控制（或由模块内部根据 PWM 极性自动判向）
编码器输出 A/B 相 → MCU 定时器正交解码输入
电池 7.4V 或 12V → 模块电源输入端
模块 5V 输出 → MSPM0G3507 供电（如果你的板子支持 5V 输入）
代码怎么写
/* mod_motor.c — 最简单的接口 */ void mod_motor_set(uint16_t pwm_l, uint16_t pwm_r) { // pwm_l: 0~1000（0=停 1000=全速正转）→ 左轮 // pwm_r: 0~1000 → 右轮 // 差速转向：pwm_l > pwm_r → 右转，pwm_l < pwm_r → 左转 bsp_pwm_set_l(pwm_l); bsp_pwm_set_r(pwm_r); } /* 为什么接口这么简单？ * 四路驱动板内部处理了方向控制和编码器调理， * 所以 MCU 侧只需要给两个 PWM 值。 * 这是"设备驱动层"的核心价值——把复杂的硬件藏起来， * 上层代码只看到 pwm_l 和 pwm_r 两个数字。 */
常见翻车
• 电机不动：先检查电池有电没、模块电源灯亮不亮。再检查 PWM 波形（示波器或逻辑分析仪看有没有方波）
• 编码器读数不动：手转轮子，串口打印编码器计数。不动 → 编码器 A/B 相接反了（或没接上拉到 VCC）
• 编码器读数跳变/负值：520 电机编码器的 A/B 相顺序和定时器正交解码的预期顺序不一致 → 交换 A 和 B
• 电机启动瞬间 MCU 复位：电机启动电流很大（堵转可达 2A+），拉低电池电压 → 如果 MCU 和电机共用一路电，加 1000μF 电容在电源输入端滤波；最好电机和 MCU 分两路供电
• 前进方向不一致：两个电机看起来都在转，但车原地转圈 → 一个电机线接反了。交换那个电机的两根电源线

MPU6050 六轴陀螺仪

角色：测航向角。白地直线段和对角线段没有黑线可巡，全靠它锁方向。
它是什么？
I2C 接口的六轴传感器（三轴加速度+三轴陀螺仪）。我们只用 Z 轴陀螺仪——不停地积分角速度，算出车转了多少度。

为什么用它？ 直线段和对角线是纯白地，灰度读不到东西。陀螺仪是唯一能告诉车"你现在指哪个方向"的传感器。

局限：陀螺仪有零漂（静止时也会慢慢往上加），需要开机校准 + 每次进对角线前重新锁航向。
怎么接线
VCC → 3.3V  |  GND → GND
SCL → PB2 (MSPM0 I2C SCL)
SDA → PB3 (MSPM0 I2C SDA)
（与 OLED 共用 I2C 总线。MPU6050 地址=0x68，OLED 地址=0x3C，不冲突）
代码怎么写
/* mod_imu.c — MPU6050 → heading */ static float gyro_bias; // 零漂（开机静止时采 1000 次取平均） static float heading; // 累计航向角 static uint32_t last_us; void mod_imu_calibrate(void) { // 车静止不动，采 1000 次 Z 轴角速度取平均 = 零漂 } float mod_imu_get_heading(void) { uint32_t now = micros(); float dt = (now - last_us) / 1000000.0f; float gz = mpu_read_z() - gyro_bias; // 减零漂 heading += gz * dt; // 积分：角速度×时间 = 角度 last_us = now; return heading; } /* 这段代码的效果： * 车原地转 90° → heading 增加 ~90 * 车走直线 → heading 不变（用于对角线锁定方向） * ⚠ 跑久了会漂移，每圈开始前重新校准 heading */
常见翻车
• 零漂没校准：车不动 heading 也会涨 → 要在每次发车前静止 1 秒采零漂
• I2C 设备地址冲突：OLED (0x3C) 和 MPU6050 (0x68) 地址不冲突，但如果地址 0x68 读不到 → 检查接线，用 I2C 扫描程序确认设备在线
• 累积漂移：跑 4 圈下来可能漂 10°+，对角线就走歪了 → 每段对角线进入时用编码器判断方向，陀螺仪只做微调

0.96 寸 OLED（I2C）

角色：调试窗口。显示当前状态、速度、里程、PID 输出。赛场必备，没有屏幕出了问题都不知道为什么。
为什么要 OLED？ 你不能一直连着串口线跑。OLED 让你在场上直接看到：现在在第几个状态、当前速度多少、编码器里程多少。出问题一眼看到哪个变量不对劲。

I2C 地址：0x3C。和 MPU6050 (0x68) 共用 I2C 总线不冲突。
代码怎么写
/* 赛场专用显示：循环显示关键信息 */ void mod_oled_show_status(State s, uint8_t lap, float dist, float speed, int16_t lpos) { oled_clear(); oled_print(0,0, "S:%d L:%d/4", s, lap); // 状态+圈数 oled_print(0,2, "D:%.0fmm", dist); // 当前段走了多远 oled_print(0,4, "V:%.0f P:%d", speed, lpos); // 速度+灰度偏移 } /* 为什么这样设计？三行显示刚好塞满 128×64 OLED： * 第1行 → 知道自己在哪（状态+圈数） * 第2行 → 知道走了多远（判断是否快到顶点） * 第3行 → 知道PID工作状态（速度+偏移量） */
常见翻车
• 不亮：I2C 地址不对（0x3C 最常见，部分模块是 0x3D）。先写 I2C 扫描程序确认地址
• 花屏：I2C 时序不对或上电没复位。初始化加 100ms 延时再发指令

蜂鸣器 + LED

角色：声光提示。题目要求每经过一个顶点要"声光提示"。
最简单的 GPIO 输出。有源蜂鸣器给高电平就响，LED 给高电平就亮。5 行代码搞定。

为什么单独做成一个模块？ 不是为了复杂——是为了习惯。每个外设一个文件，以后加功能（比如 LED 闪两下代表不同含义）只改这一个文件。
/* mod_buzzer.c — 声光提示 */ void mod_buzzer_beep(void) { BUZZER_ON(); LED_ON(); delay_ms(100); BUZZER_OFF(); LED_OFF(); } // 效果：经过顶点时响一声+闪一下，裁判知道你的车检测到了顶点
模块间的接口总结
一张表看懂谁和谁相连
  • 8 路灰度 → ADC (PA0~PA7)：8 根模拟信号线。接口 = 模拟电压
  • 四路编码器驱动 ← PWM (定时器)：2 根 PWM 信号线。接口 = 占空比
  • 四路编码器驱动 → 编码器输入 (定时器)：4 根 A/B 相数字信号线。接口 = 脉冲计数
  • MPU6050 ↔ I2C (PB2=SCL PB3=SDA)：2 根。接口 = I2C 寄存器读写
  • OLED ↔ I2C（同上）：共用总线。接口 = I2C 寄存器读写
  • 蜂鸣器/LED/按键 → GPIO：3 根数字信号线。接口 = 高低电平
