路径状态机：车怎么知道该往哪走？

你的车没有眼睛——它靠状态机告诉自己"我现在在哪一段、下一段要做什么"
为什么需要状态机？
车不是在"自由驾驶"——它在执行一个固定的序列任务。要求(3)的路径是固定的：A→C→弧线到B→D→弧线回A。

如果不用状态机，你需要在代码里写一堆 if (在A到C) { 这样走 } else if (在弧线上) { 那样走 }——很快变成一团乱麻。

状态机把"我在哪"和"我该怎么做"分开。先判断状态（我在哪一段），再根据状态决定控制策略（怎么走）。状态切换只发生一次（到顶点时），控制循环每 10ms 跑一次。

为什么 S0/S2（对角线）和 S1/S3（弧线）控制策略不同？
S0/S2 在对角线上——纯白地，没有黑线。灰度传感器读到 8 路全白 → line_pos = 0 → 方向环收不到输入。所以对角线段必须用陀螺仪锁定航向。
S1/S3 在弧线上——有黑线标记。灰度能读到偏移 → 方向环正常工作。所以要降速（防止甩出去）并加大 KP（防止纠正不够快）。
最关键的问题：怎么判断"到顶点"了？
主
编码器里程判断（所有状态通用）
每进入新状态时清零 distance。里程到了预设值（SEG_DIAG 或 SEG_ARC）就切状态。

为什么这样设计？ 场地尺寸固定，编码器精度够。对角线 A→C 的距离你是用卷尺量了写进 SEG_DIAG 的，里程到了就说明到了。这是最可靠的方法，不受光线影响。
if (distance >= seg_target) { state++; mod_encoder_reset(); mod_buzzer_beep(); }
备
灰度丢失判断（仅弧线段 S1/S3）
弧线顶点处，黑线突然没了（因为顶点就是弧线的端点，到了之后是直线段或对角线，都是白地）。连续 N 次灰度全白 → 确认到了顶点。

为什么只用于弧线段？ 对角线 S0/S2 本来就全白——从出发就是全白，不能用这个判断。这个 backup 只对 S1/S3 有效。
if (lost_count > 10) { state++; lost_count = 0; }
双重判断的必要性
  • 编码器为主：可靠，不受光线影响。但跑 4 圈累积误差可能有 5~10cm
  • 灰度丢失为 backup：在弧线顶点（S1→S2, S3→S0）可以兜底。即使编码器累积误差导致早/晚了一点，灰度突然全白也能抓住顶点
  • 只靠编码器：4 圈后误差累积 → 错过一个顶点 → 后面全错
  • 只靠灰度：对角线段出发就是全白 → 立即误判到顶点 → 车刚出发就切状态

常见翻车
❌ 状态切换了但 encoder 没清零 → 下一段起点不是 0，里程乱套。每个 case 里必须调 mod_encoder_reset()
❌ S0 和 S2 的航向角写死了 → 赛场场地角度跟实验室不完全一样。进场后先让车原地转一圈，陀螺仪测出 A→C 的实际角度，自动存到 HEADING_AC 变量
❌ S1 弧线走反了 → C→B 是顺时针走右弧，B→C 是逆时针。代码里 S1 的巡线方向要跟弧线的实际方向一致。测法：先在 S1 单独跑，看车是沿着黑线走还是反向走
❌ 第四圈计数不对 → g_lap 初始化为 0，S3→S0 切换时 lap++。从 S0 开始，lap 到 3 完成 3 圈，切换到 S0 时 lap=4 → STOP。检查 lap >= 4 还是 > 3
❌ 对角线走歪 → 陀螺仪漂移了。每进入 S0 或 S2 时，用前 100ms 锁定初始航向，之后用"当前航向 - 锁定航向"作为方向环输入
