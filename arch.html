分层架构：把代码分到正确的楼层

这么做不是为了让代码"好看"——是为了出bug时10秒定位到哪一层，换硬件只改一层
为什么必须分层？

假设你不分层，main.c 一个文件写 800 行。比赛第 3 天发现 OLED 不亮——你从哪开始查？

假设你分了层：OLED 不亮 → 直接翻到 bsp_i2c.c（板级初始化）+ mod_oled.c（设备驱动）。10 秒定位，而不是在 800 行里找。

到了赛场换了一个场地的场地尺寸：改 config/param.h 里 3 个数，不改任何逻辑代码。不分层的话你需要在一坨代码里找硬编码的数字。

换了电机驱动模块：只改 mod_motor.c 和 bsp_tim.c，app/task_control.c 一行不动。

每层详解 + 为什么要这样设计 + 常见翻车
Layer 1: bsp/ — 板级支持包
做什么：把 MSPM0 的寄存器操作封装成函数。比如 bsp_adc_read_ch(3) 返回 ADC 通道 3 的值，调用者不用知道 ADC12 有哪些寄存器。
为什么存在：如果换了一块 MCU（比如从 MSPM0G3507 换成 MSPM0L1306），只改 bsp/，上面三层完全不知道换了芯片。
怎么用 CCS SysConfig 写：在 CCS 的 SysConfig 界面里把 ADC、定时器、I2C、UART 引脚配好，点 Generate，SysConfig 自动生成 ti_driver_config.c。你的 bsp_xxx.c 直接调 DriverLib 函数就行。
❌ 常见翻车：SysConfig 配了引脚但代码里忘了调 bsp_xxx_init() → 外设不工作。每个 bsp 文件写完后立即在 main() 里 Init 并测试（串口打印一个值证明它通了）。
Layer 2: module/ — 设备驱动层
做什么：把传感器读数变成有意义的变量。灰度传感器不再是 8 个 ADC 值，而是 line_pos (-100~+100)。编码器不再是计数器，而是 distance (mm) 和 speed (mm/s)。
为什么存在：隔离"读数"和"含义"。上层代码永远不需要知道"灰度有几个传感器"——它只知道 line_pos。换成 16 路灰度也只改这一个文件。
设计原则：每个文件只暴露数据接口。比如 mod_grayscale.h 只暴露 calibrate() 和 read()，不暴露 ADC 通道号。
❌ 常见翻车：在 module 里直接操作寄存器而不是调 bsp → 换芯片时 module 也得改。module 只能调 bsp 里暴露的函数。
Layer 3: algo/ — 算法层
做什么：纯数学运算。输入两个数字（目标值 vs 实际值），输出一个数字（PWM）。
为什么存在：PID 算法不依赖任何硬件。你可以在电脑上写个 Python 版先验算参数，再复制到 MCU。不 #include 任何硬件头文件。
增量式 vs 位置式：方向环用位置式（输出 = Kp×error + Kd×delta），速度环用增量式（输出 = 上次输出 + 增量）。增量式天然防积分饱和，速度突变时不会"窜"。
❌ 常见翻车：static 变量忘了初始化 → 每次调用 PID 从随机数开始算。确认 e1[] e2[] u[] 三个 static 数组都初始化为 0。
Layer 4: app/ — 应用层
做什么：离硬件最远，离"车的行为"最近。state_machine 决定现在走哪段，task_control 在每段用不同策略控制车。
为什么存在：想改赛道逻辑？只改 app/。想从要求(2)的路径换成要求(3)的？只改 state_machine 的状态顺序。
❌ 常见翻车：状态切换条件只用一个（比如只靠编码器）→ 编码器累积误差导致错过顶点。必须双重判断：编码器为主+灰度丢失为backup。
